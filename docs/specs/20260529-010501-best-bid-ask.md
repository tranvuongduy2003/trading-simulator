# Feature: Best Bid and Ask Display
> Status: DRAFT  |  Date: 2026-05-29

## 1. Problem & Solution

**Problem:** After logging in, a trader on the main trading view cannot tell at a glance what buyers are willing to pay or what sellers are asking for **AAPL**. Without the top of the book, limit-order pricing and portfolio valuation feel disconnected from the simulated market.

**Solution:** Show the **best bid** (highest open buy price) and **best ask** (lowest open sell price) for **AAPL** on the trading screen, together with **spread** and **mid-price** (PRD FR-2.5). Values load from a market snapshot on first paint and stay current via real-time order-book updates (PRD §7.1: ≤ **500 ms** from book change to UI).

**Persona:** Aspiring Trader or Curious Learner using the local simulator (PRD §4).

**Smallest valuable version:** A compact **top-of-book** strip (best bid, best ask, spread, mid) on the authenticated trading view — **not** the full depth ladder (US-06), trade tape (US-07), or candlestick chart (US-08). Those ship in separate specs.

## 2. User Stories & Acceptance Criteria

### Story 1: See best bid and ask when I open the trading view
> As a **logged-in user**, I want to **see the current best bid and best ask for AAPL**, so that **I know the market state at a glance**.

**Happy path:**
- GIVEN I am authenticated and the simulated order book has at least one open bid at **150.2500** USD and one open ask at **150.5000** USD for **AAPL** → WHEN I land on the main trading view → THEN within **2 s** I see **Best bid** labeled **150.25** (or **150.2500** per UI formatting rules) and **Best ask** labeled **150.50**, each with an accessible text label (not color alone), and the symbol **AAPL** is visible in context (header or strip).
- GIVEN the snapshot also exposes aggregated quantity at the best level → WHEN the strip renders → THEN I may see optional size at the touch (e.g. bid size **100** shares, ask size **50** shares) without requiring the full depth table (US-06).

**Failure / edge path:**
- GIVEN I am not authenticated → WHEN I attempt to open the trading view → THEN I am redirected to login and no market prices are shown (no anonymous leak of book data on protected routes).
- GIVEN the market snapshot request fails (network or **5xx**) → WHEN the trading view loads → THEN I see a clear error state for the top-of-book strip (e.g. “Market data unavailable — Retry”) and **no fabricated** bid/ask numbers.

---

### Story 2: Best bid and ask update in real time
> As a **logged-in user**, I want **best bid and ask to update automatically**, so that **I see the market state change without refreshing the page**.

**Happy path:**
- GIVEN I am on the trading view with best bid **150.25** and best ask **150.50** → WHEN a new limit order improves the bid to **150.30** or the ask to **150.45** (place, cancel, or match affecting the touch) → THEN the strip updates to the new best prices within **500 ms** (PRD §7.1) without a full page reload.
- GIVEN I am connected to real-time market updates for **AAPL** → WHEN the order book projection changes → THEN only the top-of-book values (and derived spread/mid) change; the rest of the page does not flash or reset unrelated panels.

**Failure / edge path:**
- GIVEN the real-time connection is disconnected for more than **5 s** → WHEN I remain on the trading view → THEN I see a subtle “Reconnecting…” (or equivalent) indicator on the market strip and, on reconnect, values refresh to match the latest snapshot within **2 s**.
- GIVEN a book update message arrives with an empty bid or ask side → WHEN the UI applies it → THEN Story 3 empty/one-sided rules apply (no stale opposite-side prices from the previous message).

---

### Story 3: Spread and mid-price beside the touch
> As a **logged-in user**, I want to **see the bid–ask spread and mid-price**, so that **I understand how tight the market is and a fair reference price**.

**Happy path:**
- GIVEN best bid **150.00** and best ask **150.10** → WHEN the strip is visible → THEN I see **Spread** **0.10** USD (absolute) and **0.067%** (or equivalent rounded percentage), and **Mid** **150.05** USD (average of best bid and best ask, **NUMERIC(18,4)** precision internally).
- GIVEN spread or mid is shown → WHEN I view it → THEN numeric values use tabular/monospace styling per PRD §8.2 and include text labels (“Spread”, “Mid”) in addition to bid/ask color cues.

**Failure / edge path:**
- GIVEN only a best bid exists (no asks) → WHEN spread/mid would be undefined → THEN spread and mid show **—** or “N/A” (not **0.00** or a misleading mid).
- GIVEN only a best ask exists → WHEN spread/mid would be undefined → THEN the same **N/A** treatment applies.

---

### Story 4: Understand an empty or one-sided book
> As a **logged-in user**, I want **clear messaging when there is no bid, no ask, or no liquidity at all**, so that **I am not misled by placeholder prices**.

**Happy path:**
- GIVEN the book has both a best bid and best ask → WHEN I view the strip → THEN both sides show live prices (Stories 1–2).

**Failure / edge path:**
- GIVEN there are **no** open bids for **AAPL** → WHEN the strip loads or updates → THEN **Best bid** shows **—** or “No bids” and best ask still shows if present.
- GIVEN there are **no** open asks → WHEN the strip loads or updates → THEN **Best ask** shows **—** or “No asks” and best bid still shows if present.
- GIVEN there are **no** open bids **and** no open asks (empty book) → WHEN I view the strip → THEN both sides show empty-state copy (e.g. “No market — waiting for liquidity”) and spread/mid are **N/A**; simulated liquidity is expected to repopulate the book under normal operation (PRD §11.2).
- GIVEN the matching engine is restarting and projections are briefly stale → WHEN the API returns an empty snapshot → THEN the UI shows the empty state until the next successful snapshot or real-time update (no cached fictional prices after logout or symbol change).

## 3. Domain & Business Rules

```
BR-01: Best bid is the highest price among all open buy (bid) orders for AAPL with status Pending or Partially Filled on the book. Example: bids at 150.10, 150.25 → best bid 150.25.
BR-02: Best ask is the lowest price among all open sell (ask) orders for AAPL on the book. Example: asks at 150.50, 150.75 → best ask 150.50.
BR-03: Spread (absolute) = best ask − best bid when both exist; spread (%) = spread / mid × 100 when mid > 0. Example: bid 150.00, ask 150.10 → spread 0.10, mid 150.05, spread ≈ 0.066%.
BR-04: Mid-price = (best bid + best ask) / 2 when both exist; otherwise mid is undefined (UI shows N/A). Prices use up to 4 decimal places (PRD FR-3.1, DB money/price precision).
BR-05: Top-of-book values reflect the authoritative matching engine state after commits; read models may lag briefly but must converge within PRD §7.1 latency (≤ 500 ms) under nominal local load.
BR-06: Only AAPL is supported in MVP; requests or subscriptions for any other symbol are rejected or ignored at the product boundary (core MVP constraint).
BR-07: Quantities shown at the touch are whole-share aggregates at that price level (sum of remaining quantity for orders at the best price), not fractional shares.
```

## 4. UI Behavior **or** API Contract

### 4a. UI Behavior

```
Screen: Main trading view (authenticated)
- Arrival: User sees symbol AAPL in chrome; top-of-book strip shows Best bid, Best ask, Spread, Mid (or loading skeleton).
- Action: User navigates away and returns → strip refetches snapshot and resubscribes to market updates.
- Loading: Skeleton or “Loading market…” in the strip until first snapshot succeeds; no zeros masquerading as prices.
- Empty: “No bids” / “No asks” / “No market” per Story 4; spread/mid N/A.
- Error: Snapshot failure → inline retry on strip; does not block order form unless product chooses a global banner (optional).
- Real-time: On order book change for AAPL, update bid/ask (and derived spread/mid) within 500 ms; optional subtle highlight on change (no jarring animation per PRD §8.2).

Screen: App layout / header (optional placement)
- If top-of-book is duplicated in header later, this spec’s ACs apply to the primary trading-view strip; header last-trade price remains out of scope (Future scope).
```

Accessibility: Bid/ask labels visible in text; color is supplementary (PRD §7.5).

### 4b. API Contract

- **Endpoint(s):** `GET /api/market/orderbook?symbol=AAPL` (align with existing API prefix convention). Returns top **N** levels (default **10** per FR-2.1); client derives best bid = first/highest bid level, best ask = first/lowest ask level from sorted levels.
- **Request / response (sketch):**
  ```json
  {
    "symbol": "AAPL",
    "bids": [{ "price": 150.25, "quantity": 100, "orderCount": 2 }],
    "asks": [{ "price": 150.50, "quantity": 50, "orderCount": 1 }],
    "updatedAt": "2026-05-29T01:00:00Z"
  }
  ```
- **Errors:** `400` invalid symbol (non-AAPL); `401` no session if endpoint is authenticated; `503`/`500` with RFC 7807 body — UI shows retry. No partial fake book in error responses.
- **Auth:** Authenticated session required for MVP trading view (consistent with protected simulator routes). Public unauthenticated market data is **out of scope** unless opened in §13.
- **Idempotency:** Read-only — not required.
- **Pagination / filtering:** N/A (single symbol, fixed depth N).

## 5. Data & Storage Impact

| Concern | Answer |
|---|---|
| PostgreSQL tables / columns | **None** for new columns. Top-of-book is derived from open `orders` (partial index `ix_orders_active_book`) when rebuilding projections (DB §4, §14.2). |
| Redis keys / projections | **`orderbook:AAPL:snapshot`** — top-N levels; best bid/ask read from first levels after sort (DB §12.1). |
| Matching / channel behavior | Book changes after match/cancel/place commit → engine updates Redis snapshot → notification consumer emits market book update (Tech §7, §9). API read prefers Redis, falls back to PostgreSQL aggregation. |
| Migration needed | **No** — uses existing order book projection pattern. |
| Rebuild strategy if Redis cleared | Rebuild snapshot from PostgreSQL active orders on next engine event or on-demand query fallback (DB §12.2). |

## 6. Real-Time & Consistency

- **SignalR events:** `OrderBookUpdated` (market group `market:AAPL`) — payload includes `symbol`, `bids[]`, `asks[]` (price, quantity per level), `updatedAt`. Client recomputes best bid/ask, spread, mid from levels.
- **Read-your-writes:** Placing or cancelling an order returns before matching completes; top-of-book may update twice (acceptance then match). User sees API acknowledgment first; touch prices update when engine publishes book change (async matching invariant).
- **Stale UI handling:** On hub reconnect, refetch `GET /api/market/orderbook` and resubscribe to `market:AAPL`. Show last known values with “Reconnecting…” only if snapshot refetch also pending; do not show stale values indefinitely after logout.

## 7. Security & Privacy (MVP)

- **Authn / Authz:** Session cookie; market snapshot and hub subscription only for authenticated users on MVP trading routes. No per-user book isolation (market data is shared simulation).
- **Sensitive fields:** N/A for public top-of-book; no PII in market payloads.
- **Threat surface:** Tampering with client-side prices does not affect execution — server matching uses engine state. Rate-limiting market reads is optional for local MVP.

## 8. Observability (local MVP)

| Signal | What to emit |
|---|---|
| Structured logs | Order book snapshot served (symbol, level count, cache hit/miss); SignalR market publish; no full book dumps at info level |
| Traces | Span on snapshot query and hub subscribe (ServiceDefaults) |
| Metrics | `minimal for MVP` |
| Audit | N/A |

## 9. Edge Cases

```
EC-01: Empty book → bid/ask N/A, spread/mid N/A, copy explains waiting for liquidity.
EC-02: One-sided book → missing side N/A; spread/mid N/A.
EC-03: Best bid ≥ best ask (crossed book) → should not occur in steady state; if snapshot shows cross, display values as returned and log warning; matching engine should prevent persistent crossed resting book.
EC-04: User places limit buy above best ask → immediate match may remove touch levels; strip updates on trade/book events within 500 ms.
EC-05: Simulated liquidity provider cancels touch order → next best level becomes visible; strip steps to new price.
EC-06: Session expires while on trading view → redirect to login; strip cleared; no cached prices after auth purge.
EC-07: Redis unavailable → API falls back to PostgreSQL; slightly higher latency acceptable locally; UI still gets a consistent snapshot or error.
EC-08: Engine down / channel backlog → snapshot may be stale; after timeout, show “Market data delayed” if no update within configurable threshold (e.g. 10 s) — optional for MVP, minimum is reconnect + refetch.
```

## 10. Dependencies, Risks, Decision Triggers

- **Depends on:** US-02 (login/session), simulated liquidity (PRD §11.2) so the book is usually non-empty; matching engine and order book projection pipeline (Tech §7–§8).
- **Impacts:** US-06 (depth UI will reuse same snapshot/event), US-10–13 (order form may pre-fill from touch), US-21 (unrealized P/L uses market price — mid or last trade; last trade is separate FR-2.2).
- **External services:** PostgreSQL, Redis, Aspire-hosted API and Matching Engine.
- **Key risk:** Empty book on first launch before liquidity seeds — mitigated by simulated liquidity provider and clear empty state.
- **Decision triggers:** If top-of-book is placed in global header for all routes → ADR in `docs/memory/decisions.md`. If market data becomes public without login → ADR + security review.

## 11. Assumptions

- Best bid/ask appear on the **main trading view** in a dedicated strip or card, not only inside the future full depth panel (US-06).
- **Authenticated** users only for this MVP slice (trading view already requires login).
- Price display format: **2–4** decimal places in UI; internal **NUMERIC(18,4)**.
- Real-time transport is SignalR `OrderBookUpdated` already defined in shared contracts; no new event name required for US-05 alone.
- Optional quantity at the touch is **nice-to-have** in Story 1; spread/mid are **required** (FR-2.5).
- Last trade price in the top bar (PRD §8.1) is **not** part of US-05 (separate FR-2.2 / future story).

## 12. Out of Scope

- Full order book depth table and level aggregation UI (**US-06**).
- Trade tape (**US-07**), candlestick chart (**US-08**).
- Last trade price and daily change in header (**FR-2.2** — separate delivery).
- Multi-symbol books, fractional shares, production CDN, horizontal scale.
- Message broker, transactional outbox, guest/anonymous market pages (unless §13 resolves otherwise).
- Global MVP exclusions from `core.mdc`.
