# Feature: Trade Tape (Latest Trades)
> Status: DRAFT  |  Date: 2026-05-30
> PRD: PRD §5.2 US-07, §6.2 FR-2.3, §7.1, §7.5, §8.1–§8.3
> Tech: Tech §6.3, §8.2, §9, §10.5, §11
> DB: DB §4.7, §12.1, §14.2
> Owner: Product

## 1. Problem & Solution

**Problem:** A trader on the main trading view can see the order book (US-05, US-06) but cannot scan **recent executions** on **AAPL**. Without a trade tape, it is hard to tell whether the market is actively trading, at what prices, and in what size.

**Solution:** Display a **market-wide trade tape** — the **50** most recent **AAPL** executions (PRD FR-2.3 default), each row showing **price**, **quantity** (whole shares), and **timestamp**. The list loads from a market snapshot on first paint and **prepends** new trades in real time (newest at top) within PRD §7.1 latency (≤ **500 ms** from execution to UI).

**Persona:** Aspiring Trader or Curious Learner using the local simulator (PRD §4).

**Smallest valuable version:** A **trade tape panel** on the authenticated trading view showing **public market executions** for **AAPL** — **not** the user’s personal trade history tab (US-22 / FR-6.4), **not** the candlestick chart (US-08), and **not** the top-bar **last trade price** headline (FR-2.2; may share the same backend event later but is out of scope for this spec’s UI). Tape rows do **not** expose buyer/seller identity or user identifiers.

## 2. User Stories & Acceptance Criteria

### Story 1: See recent trades when I open the trading view
> As a **logged-in user**, I want to **see the latest AAPL trades**, so that **I can observe recent market activity at a glance**.

**Happy path:**
- GIVEN the market has at least **5** executed trades in the last hour (e.g. prices **150.20**, **150.25**, **150.30** USD with quantities **100**, **50**, **200** shares) → WHEN I land on the main trading view → THEN within **2 s** I see a **trade tape** panel titled or labeled for **AAPL** showing up to **50** rows, **newest first**.
- GIVEN a trade executed at **150.25** for **100** shares at **2026-05-30T14:00:00Z** → WHEN the tape renders that row → THEN I see **price 150.25**, **quantity 100**, and a **human-readable time** (local or labeled UTC — consistent across the panel).
- GIVEN more than **50** trades exist in history → WHEN the snapshot loads → THEN only the **50** most recent trades appear (no pagination in MVP).

**Failure / edge path:**
- GIVEN I am not authenticated → WHEN I attempt to open the trading view → THEN I am redirected to login and **no** trade rows are shown.
- GIVEN the market trades snapshot request fails (network or **5xx**) → WHEN the trading view loads → THEN the tape area shows a clear error with **Retry** (e.g. “Trades unavailable — Retry”) and **no fabricated** prices, quantities, or timestamps.

---

### Story 2: Trade tape updates in real time
> As a **logged-in user**, I want **new trades to appear at the top of the tape automatically**, so that **I see activity without refreshing the page**.

**Happy path:**
- GIVEN I am viewing the tape with **50** rows → WHEN a new **AAPL** trade executes at **150.35** for **75** shares → THEN a new row appears at the **top** within **500 ms** (PRD §7.1) and the list remains capped at **50** rows (oldest row drops off).
- GIVEN two trades execute in quick succession → WHEN both events arrive → THEN both appear at the top in **execution-time order** (newest first); no duplicate row for the same trade identifier.

**Failure / edge path:**
- GIVEN the real-time connection is disconnected for more than **5 s** → WHEN I remain on the trading view → THEN the tape shows a **Reconnecting…** indicator (or shares the market connection badge used for order book) and, on reconnect, the tape matches the latest snapshot within **2 s** (no permanent gap without refetch).
- GIVEN a real-time payload references a trade already in the list (same trade identifier) → WHEN the client applies the update → THEN the tape does **not** show duplicate rows for that trade.

---

### Story 3: Understand an empty or quiet market in the tape panel
> As a **logged-in user**, I want **clear messaging when no trades have occurred yet**, so that **I am not misled by blank or stale rows**.

**Happy path:**
- GIVEN at least one trade exists → WHEN I view the tape → THEN rows reflect live data (Stories 1–2).

**Failure / edge path:**
- GIVEN **no** trades have ever executed for **AAPL** in this environment → WHEN the tape panel loads → THEN I see empty-state copy (e.g. **“No trades yet — waiting for executions”**) and **no** placeholder price rows.
- GIVEN I log out → WHEN I log in again → THEN the tape does not flash trades from the prior session’s client cache before the fresh snapshot arrives (consistent with order book cache purge on logout).
- GIVEN portfolio reset cleared **my** trade history → WHEN I view the market tape → THEN **market-wide** trades still appear (reset does not erase public tape; PRD FR-1.4 clears **user** history only).

---

### Story 4: Trade tape fits the trading layout
> As a **logged-in user**, I want **the trade tape in the expected trading layout zone**, so that **I can read book and tape together on desktop**.

**Happy path:**
- GIVEN my viewport is at least **1280px** wide (PRD §8.3) → WHEN I open the trading view → THEN the trade tape appears in the **left panel zone** (PRD §8.1), **below** the order book depth (or below top-of-book + depth stack), with **AAPL** visible in page chrome, and numeric columns use **tabular** alignment for price and quantity (PRD §8.2).
- GIVEN the tape lists prices → WHEN I view any row → THEN price is readable without relying on color alone; optional green/red accent for “up/down vs previous row” is **supplementary** only (PRD §7.5).

**Failure / edge path:**
- GIVEN viewport width is between **768px** and **1279px** (tablet) → WHEN I open the trading view → THEN the tape remains usable in a **stacked** layout (tape below book panels) without horizontal clipping of columns.
- GIVEN the tape is loading → WHEN the first snapshot has not arrived → THEN I see a **skeleton** or “Loading trades…” in the tape zone — not a table of zeros.

## 3. Domain & Business Rules

```
BR-01: Trade tape shows market-wide executions for AAPL only (MVP single-symbol). Example: any user’s matched trade and simulated-liquidity matches appear in the same public tape.
BR-02: Default tape length N = 50 trades (PRD FR-2.3). Snapshot and client list are capped at N; older trades are not shown in the panel.
BR-03: Rows are ordered by executedAt descending (newest at top). Example: trade at 14:00:05 appears above trade at 14:00:01.
BR-04: Each row exposes price (NUMERIC(18,4)), quantity (whole shares), executedAt, and a stable trade identifier for deduplication. Buyer/seller user IDs and order IDs are not shown in the public tape (privacy).
BR-05: Tape read model reflects trades after PostgreSQL commit; UI may lag briefly but converges within PRD §7.1 (≤ 500 ms) under nominal local load (async matching: persist → engine match → projection → SignalR).
BR-06: User-scoped trade history (US-22) is a separate product surface; it must not be confused with the market tape. Portfolio reset empties my history endpoints but does not delete rows from the global trades table or market tape projection.
BR-07: Simulated liquidity executions appear in the tape like user executions; no “simulated” badge in MVP.
BR-08: Public tape is shared across all authenticated users; no per-user filtering on the market tape endpoint.
```

## 4. UI Behavior **or** API Contract

### 4a. UI Behavior

```
Screen: Main trading view — left panel (Trade tape)
- Arrival: User sees “Trades”, “Tape”, or equivalent heading; symbol AAPL; column headers such as Time, Price, Size (or Qty).
- Action: User scrolls within the tape if rows exceed viewport height (expected on desktop with 50 rows).
- Loading: Skeleton rows or “Loading trades…” until first snapshot succeeds.
- Empty: “No trades yet — waiting for executions” when snapshot returns zero trades.
- Error: Snapshot failure → inline Retry; no fake rows.
- Real-time: On each execution, prepend one row at top within 500 ms; trim to 50; optional subtle highlight on new row (no jarring animation per PRD §8.2).

Screen: Bottom panel — Trade History tab
- Unchanged scope: personal history remains US-22; tape does not replace that tab.
```

Accessibility: Column headers or labels for Time, Price, Quantity; `aria-live="polite"` on the tape container when new trades arrive (PRD §7.5).

### 4b. API Contract

- **Endpoint(s):** `GET /api/market/trades?symbol=AAPL&limit=50` (align with existing `/api` prefix and api-guidelines market trades route). Default `limit` = **50**; maximum **50** for MVP.
- **Request / response (sketch):**
  ```json
  {
    "symbol": "AAPL",
    "trades": [
      {
        "tradeIdentifier": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
        "price": 150.25,
        "quantity": 100,
        "executedAt": "2026-05-30T14:00:00Z"
      }
    ],
    "updatedAt": "2026-05-30T14:00:05Z"
  }
  ```
- **Errors:** `400` invalid symbol (non-AAPL) or `limit` out of range; `401` no session; `503`/`500` with RFC 7807 body — UI shows retry. No partial fake trades in error responses.
- **Auth:** Authenticated session required for MVP trading view (consistent with other market read routes). Public unauthenticated market data is **out of scope** unless opened in §13.
- **Idempotency:** Read-only — not required.
- **Pagination / filtering:** Not required for MVP; fixed cap **50** only.

## 5. Data & Storage Impact

| Concern | Answer |
|---|---|
| PostgreSQL tables / columns | **None** new. Reads from existing `trades` table (`symbol`, `price`, `quantity`, `executed_at`, `external_id` as public trade id) per DB §4.7; index `ix_trades_symbol_time` supports recent-by-symbol queries. |
| Redis keys / projections | **`trades:AAPL:recent`** — capped list of recent trade DTOs (DB §12.1). Written after trade commit; read preferred by query handler with PostgreSQL fallback. |
| Matching / channel behavior | After `ProcessMatch` commits → engine/appends to Redis list → notification consumer emits `TradeTapeEntryPublished` to `market:AAPL` (Tech §7, §9). API read does not block on match. |
| Migration needed | **No** — uses existing `trades` table and Redis key pattern. |
| Rebuild strategy if Redis cleared | Rebuild `trades:{symbol}:recent` from PostgreSQL `ORDER BY executed_at DESC LIMIT 50` on next trade event or on-demand query fallback (DB §12.2). |

## 6. Real-Time & Consistency

- **SignalR events:** `TradeTapeEntryPublished` (market group `market:AAPL`) — payload includes `symbol`, `tradeIdentifier`, `price`, `quantity`, `executedAt` (product-level shape already in shared contracts). Client prepends one row and trims to **50**.
- **Related (out of UI scope here):** `LastTradePriceChanged` may fire on the same execution for future top-bar work (FR-2.2); tape implementation should not depend on the user seeing the header widget.
- **Read-your-writes:** Placing an order returns before matching completes; the user’s trade may appear on the tape only after the engine executes, consistent with async matching.
- **Stale UI handling:** On hub reconnect, refetch `GET /api/market/trades` and resubscribe to `market:AAPL`. Do not show stale tape indefinitely after logout.

## 7. Security & Privacy (MVP)

- **Authn / Authz:** Session cookie; market tape and hub subscription for authenticated users on MVP trading routes. Tape payload must not include other users’ identifiers.
- **Sensitive fields:** No PII in market tape messages; trade identifiers are opaque UUIDs.
- **Threat surface:** Client-side tape manipulation does not affect execution; server matching uses authoritative state.

## 8. Observability (local MVP)

| Signal | What to emit |
|---|---|
| Structured logs | Market trades snapshot served (symbol, count, cache hit/miss); SignalR tape publish; no full user ids in market payloads |
| Traces | Span on snapshot query and hub subscribe (ServiceDefaults) |
| Metrics | `minimal for MVP` |
| Audit | N/A |

## 9. Edge Cases

```
EC-01: No trades yet → empty state; no zeros as prices.
EC-02: Exactly 50 trades in Redis → snapshot returns 50; new trade prepends and drops oldest.
EC-03: Duplicate SignalR delivery → client dedupes by tradeIdentifier.
EC-04: Redis unavailable → API falls back to PostgreSQL; acceptable higher latency locally; UI gets consistent snapshot or error.
EC-05: Engine down / channel backlog → tape may pause; reconnect + refetch restores within 2 s.
EC-06: Session expires on trading view → redirect to login; tape cleared with market cache purge.
EC-07: Non-AAPL symbol query → 400; UI only offers AAPL in MVP.
EC-08: Trade at price 150.1234 → display up to 4 decimal places; internal NUMERIC(18,4).
EC-09: User reset → GET /api/trades (my history) empty; market tape unchanged.
EC-10: Simulated liquidity trade → appears on tape same as user trade (BR-07).
```

## 10. Dependencies, Risks, Decision Triggers

- **Depends on:** US-02 (login/session); matching engine persisting trades and trade projection pipeline (Tech §7–§10); simulated liquidity (PRD §11.2) so tape is usually non-empty after warm-up.
- **Impacts:** US-08 (chart may consume same trade stream later); US-09 (real-time without refresh — partially satisfied); US-22 (personal history must stay distinct); FR-2.2 last trade in top bar (separate UI, may share events).
- **External services:** PostgreSQL, Redis, Aspire-hosted API and Matching Engine.
- **Key risk:** `GET /api/market/trades` and Redis tape projection are **not** fully wired in the codebase yet (SignalR message types exist; publisher calls are not yet emitted from match path). Delivery requires end-to-end slice in `/plan`.
- **Decision triggers:** If tape moves to center panel or global header → ADR in `docs/memory/decisions.md`. If market data becomes public without login → ADR + security review.

## 11. Assumptions

- Trade tape is **authenticated** on the main trading view (same as order book reads).
- Placement is **left column below order book depth** until full PRD §8.1 grid (chart center, order form right) lands in later specs.
- Row fields are **time, price, quantity** only — no aggressor side column unless added via §13.
- **50** is both default and maximum `limit` for MVP.
- Hub method name remains **`TradeTapeEntryPublished`** (existing contracts).
- Personal **Trade History** tab continues to use **`GET /api/trades`** (owner-scoped, paginated) — different from market tape.

## 12. Out of Scope

- Candlestick chart and OHLC aggregation (**US-08**).
- Top-bar **last trade price** and daily change display (**FR-2.2** — separate story/spec).
- User **trade history** tab UX and filters (**US-22** / FR-6.4) — only called out to avoid confusion.
- Showing **buyer/seller** or **my trade** highlighting in the public tape.
- Pagination, date filters, export, and CSV download on market tape.
- Multi-symbol tapes, fractional shares, production CDN, horizontal scale.
- Message broker, transactional outbox, broker integration.
