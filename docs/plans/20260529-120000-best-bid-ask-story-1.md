## Executive summary

Deliver **Story 1** of US-05: authenticated users see **best bid** and **best ask** for **AAPL** on the main trading view within **2 s** of landing, loaded from `GET /api/market/orderbook?symbol=AAPL`. The read path follows Tech §10.5: **Redis snapshot first** (`orderbook:AAPL:snapshot`), **PostgreSQL aggregation fallback** from active orders (`ix_orders_active_book`). The matching engine worker is still a stub (no in-memory book or Redis writer yet), so **PostgreSQL fallback is required** for local demos and integration tests until engine projections land in a later epic slice.

Frontend adds a compact **top-of-book strip** on `trading-page` (not global header), with loading skeleton, labeled bid/ask (text + supplementary color), optional touch size, and a retryable error state—**no fabricated prices**. SignalR live updates, spread/mid, and empty-book marketing copy stay in Stories 2–4.

## Goals and non-goals

**Goals**

- G1: `GET /api/market/orderbook?symbol=AAPL` returns sorted top-N bid/ask levels for authenticated sessions (RFC 7807 on errors; `400` for non-AAPL).
- G2: Trading view shows **Best bid**, **Best ask**, **AAPL** context, optional quantities; loads via TanStack Query on first paint.
- G3: Prove AC with Api integration tests (seeded orders) and a manual Aspire checklist.

**Non-goals** (this plan will not do)

- NG1: SignalR `OrderBookUpdated` client handling or ≤500 ms live refresh (**Story 2**, #59).
- NG2: Spread, mid-price, or percentage display (**Story 3**, #60).
- NG3: Empty/one-sided book copy (“No bids”, “No market”) (**Story 4**, #61).
- NG4: Matching engine loop, simulated liquidity host, `PlaceOrder`, or Redis snapshot **writes**.
- NG5: Full depth table (US-06), trade tape, chart, last-trade header.

## Data & migration plan

| Artifact | Action | DB reference |
|----------|--------|--------------|
| EF migration | None | — |
| Redis keys | Read-only `orderbook:AAPL:snapshot` | DB §12.1 |
| Book recovery | N/A — PG fallback builds levels from open orders | DB §12.2, §14.2 |

**PostgreSQL aggregation (fallback):** Group active orders by `symbol`, `side`, `price`; sum `original_quantity - filled_quantity` as level quantity; count orders per level; sort bids **desc** by price, asks **asc**; take top N (default 10). Best bid = first bid level; best ask = first ask level (BR-01, BR-02, BR-07).

**Redis snapshot (preferred when present):** JSON aligned with Contracts response shape; same sort semantics as spec §4b. If key missing or Redis errors, fall back to PG without failing the request (EC-07).

## Authorization, session, and domain notes

- **Session model:** Cookie session; `ICurrentUserAccessor` in handler returns `401` if missing (same as `GetMyWallet`).
- **Route protection:** Trading route already behind `ProtectedRoute` — no market fetch when `authStatus !== 'authenticated'`.
- **Domain rules (implementers must not violate):**
  - BR-01/02: Best prices from **highest bid** / **lowest ask** among active book orders.
  - BR-06: Reject or ignore non-`AAPL` symbol at API boundary (`400` + stable `code`).
  - BR-07: Level quantity = sum of **remaining** whole shares at that price.
  - Never return placeholder bid/ask on `5xx` or empty error body.

## Implementation details (for /build)

- **Query:** `GetOrderBookSnapshotQuery(string Symbol, int Depth = 10)` → `Result<OrderBookSnapshotResponse>`.
- **Errors:** `UNAUTHORIZED` (401); `INVALID_SYMBOL` (400) when `symbol != "AAPL"` (case-normalize to uppercase).
- **Handler flow:** authorize → validate symbol → `readRepository.GetSnapshotAsync(symbol, depth)` → map to Contracts.
- **Redis JSON shape:** Match HTTP response body for simplicity (engine writer in future epic should use same shape).
- **Frontend API:** `apiClient.get<OrderBookSnapshotResponse>('/api/market/orderbook', { params: { symbol: 'AAPL' } })` in `features/market/api.ts`.
- **Do not** wire `onOrderBookUpdated` to strip in this plan — invalidation already exists for Story 2.
