## Executive summary

Deliver **Story 2** of US-05: the top-of-book strip on the trading view updates **without a full page reload** when the order book projection changes, via SignalR `OrderBookUpdated` on group `market:AAPL`. Story 1 already provides the snapshot API, hub client, market subscription on login, and query invalidation on hub messages; this plan adds a **canonical server publish path** (reload snapshot from the read model, then fan-out) and upgrades the client to **apply hub payloads directly** (`setQueryData`) plus **reconnect UX** (5 s вҖңReconnectingвҖҰвҖқ, refetch within 2 s on reconnect).

The matching engine loop and Outgoing Trade Event Channel consumer (Tech В§7, В§9.3) are **not implemented yet** вҖ” only `ResetPortfolio` publishes `OrderBookUpdated` today (with an empty book payload). Story 2 wires a reusable `IOrderBookRealtimeProjection` service and documents the hook for future `PlaceOrder` / `CancelOrder` / `ProcessMatch` handlers. Automated proof of вҖңbid improves to 150.30вҖқ uses an integration test that seeds orders and invokes the projection service; full manual happy-path via order placement waits on the orders epic.

## Goals and non-goals

**Goals**

- G1: After any in-scope book mutation, Api publishes `OrderBookUpdated` with levels from `IOrderBookSnapshotReadRepository` (Redis preferred, PostgreSQL fallback вҖ” same as snapshot GET).
- G2: Authenticated clients subscribed to `market:AAPL` update bid/ask in the strip within **500 ms** of the hub message (no full-page reload; prefer `setQueryData` over refetch-on-invalidate).
- G3: On hub disconnect **> 5 s**, show a subtle вҖңReconnectingвҖҰвҖқ on the strip; on reconnect, resubscribe + refetch snapshot within **2 s** (spec В§13 Q3: keep last prices until refetch fails).
- G4: One-sided hub payloads clear the missing side (no stale opposite price) вҖ” aligns with Story 3/4 rules using existing `deriveTopOfBook`.
- G5: Api integration test: SignalR client receives `OrderBookUpdated` after projection publish following DB seed change.

**Non-goals** (this plan will not do)

- NG1: Matching engine worker, incoming/outgoing channels, simulated liquidity (**separate epic**).
- NG2: Initial snapshot load, spread/mid labels, empty-book marketing copy (**Stories 1, 3, 4** вҖ” #58, #60, #61).
- NG3: `POST /orders` / cancel pipeline (hook only; no order commands in repo today).
- NG4: Redis snapshot **writes** from engine (read + publish-from-read in Api only).
- NG5: SignalR backplane, multi-symbol, frontend automated tests.
