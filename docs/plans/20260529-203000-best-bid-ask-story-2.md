## Executive summary

Deliver **Story 2** of US-05: after Story 1’s HTTP snapshot, the top-of-book strip **updates in place** when the order book projection changes, via SignalR `OrderBookUpdated` on group `market:AAPL`, without reloading the trading page. The client applies hub payloads directly to the TanStack Query cache (`setQueryData`) for sub-500 ms perceived latency (PRD §7.1), shows a **Reconnecting…** badge after **5 s** hub disconnect while keeping last prices (spec §13 Q3), and **refetches** the snapshot on reconnect within **2 s**.

The matching engine loop and Redis **writers** remain out of scope; this plan adds an **Application-level publish skeleton** (`IOrderBookMarketDataNotifier`) that reads the current snapshot (Redis/PG via existing read port) and fans out SignalR—callable from integration tests now and from the engine notification consumer later. Spread/mid UI and marketing empty-state copy stay in Stories 3–4.

## Goals and non-goals

**Goals**

- G1: Authenticated sessions subscribe to `market:AAPL` (already wired in `useSimulationHub`) and receive `OrderBookUpdated` when the server publishes.
- G2: `OrderBookUpdated` updates `['market','orderbook','AAPL']` cache without invalidating wallet/portfolio panels.
- G3: Reconnect UX: resubscribe + snapshot refetch; strip badge after 5 s disconnected.
- G4: Api integration test proves hub delivers a touch price change to a subscribed client.
- G5: Manual Aspire checklist for live strip update and reconnect.

**Non-goals** (this plan will not do)

- NG1: Matching engine channel loop, simulated liquidity, or automatic publish on every place/cancel/match (**deferred** to matching-engine epic; hook documented in Task 5).
- NG2: Spread, mid-price, or percentage labels (**Story 3**, #60).
- NG3: “No bids” / “No market” marketing copy (**Story 4**, #61).
- NG4: Redis snapshot **write** path in MatchingEngine (only read + publish skeleton here).
- NG5: Full depth table, trade tape, chart, last-trade header.

## Authorization, session, and domain notes

- **Session model:** Hub uses cookie auth (`withCredentials: true`); `SubscribeToMarket` does not require user id (market group is shared).
- **Route protection:** Trading page remains behind `ProtectedRoute`; hub starts only when `authStatus === 'authenticated'`.
- **Domain rules:** BR-01/02 enforced server-side in snapshot read path; client displays first bid/ask level only. BR-05: client applies hub message immediately; server publish reads latest projection.
- **One-sided updates:** Replacing full snapshot in cache clears missing sides (no stale opposite price from prior message).

## Implementation details (for /build)

- **Notifier:** `OrderBookMarketDataNotifier(IOrderBookSnapshotReadRepository, IRealtimeNotificationPublisher)` — map `OrderBookLevelReadModel` → `OrderBookLevelMessage` (price, quantity only).
- **Symbol guard:** Only `AAPL`; no-op or throw for others (match BR-06).
- **Frontend query key:** `['market','orderbook','AAPL']` (must match `useOrderBookQuery`).
- **Hub method names:** Client `OrderBookUpdated` per `simulationHubClientMethods.orderBookUpdated` in `web/src/types/realtime.ts`.
- **Reconnect:** `queryClient.invalidateQueries({ queryKey: ['market', 'orderbook', 'AAPL'] })` on reconnect only; live updates use `setQueryData`.
- **500 ms AC:** Measured locally with hub push + React update; no server timer required for MVP.
- **Engine hook (future):** After `orderbook:{symbol}:snapshot` Redis SET in match processor, call `IOrderBookMarketDataNotifier.NotifyOrderBookChangedAsync(symbol)`.
